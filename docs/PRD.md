# munhak-board PRD

> 문학 공모전 모아보기 대시보드 — 요구사항 정의서
> 상태: **초안 (v0.1)** · 표시된 ❓ / `[결정필요]` 항목은 함께 채워나간다.

---

## 1. 배경 & 문제

시·소설·수필·백일장 등 문학 공모전 공고가 여러 사이트(위비티·엽서시·올콘·지자체·기관 홈페이지)에 흩어져 있어, 매번 여러 곳을 돌며 확인해야 한다. 마감을 놓치기도 쉽다.

**목표:** 흩어진 문학 공모전 공고를 한 화면에 모아, **마감 임박순으로 한눈에** 보고 놓치지 않게 한다.

## 2. 목표 & 비목표

**목표 (이번 범위)**
- 주요 출처를 **주 1회 자동 수집**해 한 곳에 통합
- **마감 임박순** 정렬, **장르별 필터**(시/소설/수필/희곡/동화/백일장), 검색
- 서버·DB·비용 없이 운영 (GitHub Actions + GitHub Pages)
- 모바일에서도 보기 좋게

**비목표 (지금은 안 함)**
- 공모전 응모/접수 대행
- 회원·로그인·즐겨찾기 (※ 향후 후보)
- 실시간(매일/매시간) 수집 — 주 1회로 충분
- 모든 군소 출처 100% 커버 — 핵심 출처 우선, 니치는 점진 추가

## 3. 사용자 & 시나리오

- **주 사용자:** 본인(글 쓰는 사람 / 공모전 참가 희망자)
- 시나리오:
  1. 주말에 대시보드 열어 **이번 주 새로 뜬 공고** 훑기
  2. **"시"만** 필터해서 마감 가까운 것부터 확인
  3. 관심 있는 공고 클릭 → 원본 사이트로 이동해 상세 확인
- ❓ 나 혼자 보는 용도인가, 남에게 공유/공개도 하나? → `[결정필요]` (Pages는 기본 공개 URL)

## 5. 데이터 모델 (`contests.json`)

각 공모전 1건의 필드:

| 필드 | 타입 | 설명 |
|---|---|---|
| `title` | string | 공모전명 |
| `url` | string | 원본 공고 링크 |
| `organizer` | string | 주최사 |
| `genres` | string[] | 장르 태그 (아래 6번 규칙) |
| `deadline` | string\|null | 마감일 `YYYY-MM-DD` (없으면 null) |
| `dday` | string | 표시용 D-day (예: `D-9`) |
| `status` | string | 접수중 / 접수예정 / 모집 등 |
| `summary` | string | 한 줄 요약 (엽서시 제공분) |
| `prize` | string | 상금 (엽서시 상세 best-effort, 없으면 "") |
| `eligibility` | string | 응모자격 (엽서시 목록, 없으면 "") |
| `sources` | string[] | 출처 (중복 병합 시 복수) |

> **위비티 상금/자격 미수집:** 위비티 상세페이지는 봇 차단(헤드리스 브라우저로도 본문 미렌더)이라
> 상금·자격을 못 받는다. 위비티 건은 두 필드가 빈 값이며 원문 링크로 확인을 위임한다.
> Playwright는 위비티에 무효해 **CI 의존성에서 제외**(requirements.txt 미포함).

최상위: `{ generated_at, count, contests[] }`

- **중복 제거:** 제목에서 회차·연도·기호 제거 후 정규화 키로 병합. 마감일 명시된 쪽 우선.
- **D-day 신선도:** 수집은 주 1회지만, 대시보드가 `deadline`으로 **매번 D-day를 다시 계산**해 표시 → 날짜 안 틀어짐. `[결정필요: 이 방식 OK?]`

## 6. 장르 분류 규칙

제목+요약 키워드 매칭으로 태그 부여(복수 가능). 매칭 없으면 `기타`.

| 장르 | 키워드(예) |
|---|---|
| 시 | 시, 시조, 동시, 창작시, 디카시, 엽서시 |
| 소설 | 소설, 장편, 단편, 콩트, 웹소설, 앤솔로지 |
| 수필 | 수필, 에세이, 산문, 수기, 편지, 독후감, 서평 |
| 동화/아동 | 동화, 아동문학, 어린이, 청소년문학, 그림책 |
| 희곡/극본 | 희곡, 시나리오, 극본, 대본, 드라마 |
| 백일장 | 백일장, 글짓기 |

- ❓ 분류가 거칠다(예: "기타" 7건). 정확도 더 높일지 `[결정필요]`

## 7. 기능 요구사항

**F1. 수집기 (완료)**
- 주 1회 위비티·엽서시 파싱 → 병합/중복제거/마감필터 → `contests.json` 갱신
- 한 출처가 깨져도 나머지는 살림

**F2. 대시보드 (PRD 확정 후 작성)**
- `contests.json` 읽어 카드/리스트로 표시
- **정렬:** 마감 임박순 (기본)
- **필터:** 장르 탭 (전체/시/소설/수필/희곡/동화/백일장)
- **검색:** 제목·주최사 키워드
- 각 항목: 제목, D-day(실시간 계산), 마감일, 주최사, 장르, 출처, 원본 링크
- 모바일 반응형
- ❓ 추가로 보고 싶은 것: 상금? 응모자격? → `[결정필요]`

**F3. 자동화 (예정)**
- GitHub Actions cron 주 1회(❓ 요일/시각 `[결정필요]` — 예: 월 09:00 KST)
- 수집 후 `contests.json` 자동 커밋 → Pages 자동 반영

## 8. 비기능 요구사항

- **비용 0원** (Actions 무료 한도 + Pages 무료)
- **서버리스** — 상시 실행 인프라 없음
- 출처 사이트 부담 최소화(주 1회, 소량 페이지)
- 데이터 부정확 가능성 고지(원본 확인 권장 문구)

## 9. 향후 후보 (백로그)

- 올콘 등 출처 추가 (SSL 우회)
- 마감 임박 **알림**(메일/카톡) — `[관심 있으면 우선순위 ↑]`
- 즐겨찾기/봤음 표시
- 지자체·지하철 시 공모 같은 니치 소스 직접 수집

## 10. 결정사항 (v0.2 확정분)

1. **공개 범위** — ✅ 지금은 Pages 미설정. 대시보드 파일만 만들어두고 **사용자가 직접 배포**.
2. **cron 시각** — ✅ **금요일 18:00 KST** (= cron `0 9 * * 5` UTC).
3. **대시보드 표시 항목** — ✅ 기본 세트 + **응모자격 + 상금**. 상금은 목록에 없어 **상세페이지 수집 추가**.
4. **장르 정확도** — ⏳ 현 수준 유지, 추후 개선.
5. **알림** — ✅ 이번엔 안 함. 데이터 안정화 후 백로그.
6. **수집량** — ✅ 출처별 2페이지 유지.

### v0.3 — 대시보드 스택 (확정)

- **프레임워크:** Next.js 15 (App Router) + TypeScript, **정적 export**.
- **스타일:** Tailwind v4 + CSS변수 **2층 토큰**(primitive→semantic). 디자인 시스템은 `web/DESIGN.md`.
- **무드:** 모던 한국 에디토리얼(WIRED 구조 + Pretendard). 크림 배경 · 4px 카드 · 괘선 · 잉크블루 링크 · 버밀리언 강조.
- **D-day 색:** ≤5일 빨강 / 6~14일 황토 / 15일+ 초록 / 미정 회색.
- **데이터:** `web/public/contests.json` 을 런타임 fetch(주1회 갱신, 재빌드 불필요).
- **호스팅:** Vercel(`web/` 루트). 폴더는 같은 repo 모노레포.
- **컴포넌트:** Atomic (atoms·molecules·organisms).
- 기존 바닐라 `docs/index.html` 은 폐기.
